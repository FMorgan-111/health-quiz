# AI 使用复盘

这个项目（BMI 健康评估漏斗）基本是我带着 AI 编码助手（Claude Code、Codex）一起做下来的。与其堆一份"AI 帮我干了 1234"的清单，我更想老实讲一遍整个过程：怎么从搭框架开始、中间在哪儿栽了跟头、算法和测试是怎么一点点磨出来的，以及几次我没听 AI 的。

## 一、搭框架：第一版直接搭歪了

开局很顺：把 TASK.md 的四阶段需求喂给 AI，开了 Supabase，让它起 Prisma schema，顺手把带 Postgres service 的 GitHub Actions 也配上了——这些 AI 做得又快又对，没什么好说的。

问题出在框架的"形状"上。AI（Codex 和早期的 Claude 都一样）读完 TASK.md，非常自信地把这东西建成了一个 **likert 问卷打分引擎**：七张规范化的问卷表（questionnaire / question / option / assessment…）、一套问卷→维度得分的算法、外加 JWT 注册登录。代码写得挺漂亮，schema 规范、测试还全绿。

但方向是反的。TASK.md §1、§2 要的是 **BMI 健康评估漏斗**——填性别、目标、身高体重，算出 BMI、建议热量、目标日期和预测曲线。压根没有"问卷""维度打分"这回事，鉴权那段还白纸黑字写着"不要求注册登录，用匿名 cookie 会话"。

是我对着 TASK.md 一条条核，才发现 AI 把需求读拧了。然后做了整个项目最大的一个决定：**推翻重来。** schema 从七张表砍回三张（sessions / results / subscriptions），JWT 那套全删、换成匿名 cookie 会话，打分逻辑整个扔掉、重写成 BMI 算法。git 里那个 `feat: rework to BMI health assessment per TASK.md` 就是这个转折点。

> 这事给我的教训挺实在：**AI 越自信、产出越完整、测试越绿，越得回到需求本身去对。** 测试全绿只能证明"它自己跟自己自洽"，证明不了"做的是对的东西"。后面我对 AI 的每一段产出都多了一道"这真是 TASK 要的吗"的检查。

## 二、数据库：返工之后才定下来的三张表

重建之后的 schema 不是照着 TASK.md §4 抄的，几个关键点都是我跟 AI 来回掰扯定下来的：

- 录入字段**全部 nullable**——分步保存嘛，用户填到第三步关页面，没填的就该是 `null`，不能用占位值糊弄；
- 加一个 `version` 整型做**乐观锁**，配 `updateMany({ where: { id, version } })`，专门挡并发和乱序提交；
- `target_date` / `projection_curve` 这两个会员专属字段，我让它放在 `results` 表里、由 **API 层**决定要不要返回，而不是在 schema 层面去区分会员——schema 不该知道"会员"这个概念；
- 连接串拆成 pooler（运行时）和 direct（迁移）两套——这是 AI 提醒我的，Supabase + Vercel serverless 不这么配迁移会出问题。

`users` 关联我让它留了个 nullable 的预留字段但没启用——这期不做登录，但留个口子，将来要接也不至于改表。这种"预留但不过度设计"的分寸，是我明确叫 AI 别画蛇添足才守住的。

老实说几个还没打磨到位的地方也记一下：`bmi_category` 现在存的是中文文本（"超重"），更专业的做法是存 enum key、展示层再翻译；数值边界目前只在 Zod 应用层把关，DB 层没补 CHECK 约束。这些是已知的、留给下一轮的债，不是没看见。

## 三、算法：纯函数 + 把 now 注进去

健康评估算法（`lib/health/compute.ts`）是 AI 写的，内容是常规的那套：BMI 加 WHO 分类、Mifflin-St Jeor 基础代谢（性别 `other` 取男女均值、乘活动系数、再按目标加减热量）、用安全周变化推目标日期、逐周线性逼近画预测曲线。

但有两处是我要求改的，也是后面测试能写扎实的前提：

- **写成纯函数，并且把 `now` 当参数传进去**，而不是函数里直接 `new Date()`。就这一下，算法彻底可测了——给定输入和时间，输出是确定的，目标日期能精确断言，不用跟系统时钟较劲。
- **脱敏放在序列化层**（`lib/health/result-view.ts`）：非会员的响应里，受保护字段是**根本不存在**，不是给个 `null` 或者前端藏起来。`toPublicResult` / `toFullResult` 两个函数把这条钉死。

乐观锁、还有 goal 跟目标体重的跨字段一致性校验（想减重却把目标填得比现在还重，直接拦），也都是 AI 落的代码、我定的边界。

## 四、测试：让 AI 生成，但边界是我对着 TASK 抠出来的

测试这块我用 AI 用得最狠，但思路不是"你看着写几个"，而是我对着 TASK.md §4 一条条列出要覆盖的场景，再让 AI 落成用例。最后是六个套件、六十多个 case，分四层：

- **算法单元**（`compute.test.ts`）：WHO 阈值边界、热量公式、目标日期和曲线方向，外加一组极端但合法的输入——250cm/25kg、150cm/400kg、年龄 13 和 120、目标体重跨度拉满，验证结果有限、曲线收敛不越界。
- **校验单元**（`validation.test.ts`）：每一步 Zod 的越界、非法、缺失、类型错（`NaN`/`Infinity`/小数/字符串/`null`），enum 白名单，还有跨字段一致性。
- **持久化集成**（`persistence.test.ts`）：直接打真库——分步写入、进度恢复、乐观锁并发只有一个赢、过期 version 命中零行、级联删除。
- **流程 + 端到端**（`step-flow.test.ts` / `pay-e2e.test.ts`）：走真实 handler，跳步拦截、回退重提、重复提交幂等、版本冲突 409；以及脱敏 → `/pay` → 解锁的全链路。

这里有个刻意的选择：集成测试**不 mock Prisma**，直接连 Postgres。因为持久化、事务、乐观锁、外键级联本身就是要验证的东西，把 DB mock 掉之后测出来的"绿"没有意义。AI 一开始想用内存 mock 图快，被我否了。

还有一类是 AI 自己漏的。"乱序/重复提交"最初那版测试没覆盖，我对照 §4 发现缺了，才补了一整套经真实 handler 的 `step-flow`。所以测试覆盖到这个程度，不是 AI 一次到位，是"我核 TASK → 发现缺口 → 让它补"来回了好几轮。

## 五、几次我没听 AI 的

**最大的一次就是第一节那个方向性返工**，不重复了——那是整个项目里我觉得最值的一次"不顺着 AI 走"。

另一次小但典型，是个**测试断言本身就错**的坑。`/pay` 的端到端测试，AI 最初的写法是：拿落库的 `computed_at` 当 `now`，重跑一遍 `compute`，再拿结果去比对接口返回的 `target_date`。听着挺合理，其实是错的——算的时候用的是 `new Date()`，`computed_at` 是 Postgres 自己 `now()` 落的，两个时钟差几百毫秒，断言会偶发挂。我让它改成**直接断言"接口返回的解锁字段 == 数据库里存的那一行"**。这才是真正要保的不变量：会员看到的，就是当初算好存下的那份，没有第二次计算、没有伪造。

说到底，AI 在这个项目里是个很强的执行者：schema、算法、测试、CI 它都能又快又好地落地。但"做的是不是对的东西""这个断言到底在保什么""哪个场景还没覆盖"——这些得我来盯。它负责快，我负责对。
