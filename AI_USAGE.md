# AI 使用复盘

这个 BMI 健康评估漏斗是我用 AI 编码助手（主要是 Claude Code 和 Codex）写出来的。下面老实记一下 AI 在各个环节帮了什么忙、哪些地方是我自己拍板的，以及一次我把 AI 整个方案推翻重来的经过。

## 数据库建模

表结构这块，我让 AI 照着 TASK.md 第四节的要求把 Prisma schema 搭起来：`sessions`、`results`、`subscriptions` 三张表，UUID 主键，1:1 关联，级联删除。但几个关键决定不是它一句话给的，是我俩来回确认才定下来的。

录入字段我要求全部 nullable，因为是分步保存，用户填到哪算哪，没填的就是 null。`version` 那个乐观锁字段也是我特意加的，配合 `updateMany({ where: { id, version } })` 来挡并发和乱序提交。还有受保护字段（`target_date`、`projection_curve`）的位置——我没让它在 schema 里区分会员，而是放在 `results` 表里，到底返不返回交给 API 层去判断。连接串拆成 pooler（跑的时候用）和 direct（迁移用）两套，这个是 AI 提醒我的，Supabase 配 Vercel serverless 确实得这么搞。

说句实话，最早 AI 把这块建成了七张表的"问卷打分引擎"，整个方向是错的，后面会讲到。

## 关于 Mock 数据

这个项目我刻意没怎么用 mock。最主要的一个取舍是：集成测试直接连真的 Postgres，不去 mock 掉 Prisma。道理很简单，持久化、事务、乐观锁、外键级联这些本来就是要验证的东西，你把它 mock 掉，测出来的"绿"等于没测。

真正用了替身的只有两处。一是单元测试里手搓的几个输入 fixture，比如 `compute` 的 `input()` helper、`result-view` 的 `row`，纯数据没有行为。二是端到端测试里用 `vi.mock("next/headers")` 把 cookie 那层接到真库里预先建好的 session 上——这里替掉的是浏览器环境，不是数据本身。

我也没引入那种随机批量造假数据的东西。整个测评就七个结构化字段，按场景一个个手写比生成器更可控，断言也能写得更准。

## 复杂逻辑

健康评估算法（`lib/health/compute.ts`）是 AI 写的：BMI 加 WHO 分类，Mifflin-St Jeor 算基础代谢（`other` 性别取男女均值，再乘活动系数、叠加目标增减的热量），按每周安全变化量推目标日期，再用逐周线性逼近画出预测曲线。这里有个我坚持要它改的点——把它写成纯函数、把 `now` 当参数传进去，这样才好测。

服务端脱敏那块（`lib/health/result-view.ts`）也是，我让它写成 `toPublicResult` 和 `toFullResult` 两个序列化函数，非会员的响应里受保护字段是"压根没有"，不是给个 null。乐观锁和 goal 跟目标体重的跨字段一致性校验也都是 AI 落的代码，但边界条件是我划的。

## 测试

测试是 AI 按需求分层生成的，边界场景我对着 TASK.md 第四节一条条核对补的。单元测试覆盖 compute 的 WHO 阈值边界，还有那些极端但合法的输入，比如 250cm 配 25kg、年龄 13 或 120、目标体重跨度特别大；脱敏部分专门断言受保护字段"不存在"。校验测试覆盖每一步 Zod 的越界、非法、缺失、类型错（`NaN`、`Infinity`、小数、字符串、`null` 都试了）、enum 白名单和跨字段一致性。集成测试打真库，跑分步保存加进度恢复、乐观锁并发、乱序重复回退提交、级联删除。端到端就是脱敏 → `/pay` → 解锁这一整条链路。

举个例子，"乱序/重复提交"这组一开始是漏的，我对着第四节才发现，补了一套走真实 handler 的 `step-flow` 测试。

## 一次我把 AI 推翻重来

最严重的一次是方向性的：AI 把整个产品做错了。

早期那版 AI（Codex 和早期的 Claude 都有份）读完 TASK.md，特别自信地把项目做成了一个"likert 问卷打分引擎"——七张规范化的问卷表、一套问卷到维度分数的打分逻辑，还配了 JWT 注册登录。代码质量不差，测试也全绿。问题是方向整个反了：TASK.md 第一、二节要的是 BMI 健康评估漏斗（性别、目标、身高体重这些，算出 BMI、热量、目标日期、曲线），根本不是问卷。

是我对着 TASK.md 才发现它把需求读拧了，于是叫停、要求整体返工到 BMI 模型：schema 从七张表重建成三张，去掉 JWT 换成 cookie 匿名会话，计算逻辑和前端全部重写。

这事给我的教训是：AI 越自信、东西做得越完整，越得回到需求本身去核对，别被"它跑起来了"牵着走。测试全绿只能说明实现自洽，证明不了你做的是对的东西。

还有一次是测试断言本身写错了。`/pay` 的端到端测试最早是这么写的：拿落库的 `computedAt` 当 `now`，重新跑一遍 `compute`，再去比对解锁后的 `target_date`。但 compute 当时用的是 `new Date()`，而 `computedAt` 是 Postgres 用 `@default(now())` 落的，两个时钟差个几百毫秒，断言就会偶尔挂。发现之后我把这种"拿两个不同时刻互相验证"的思路否了，改成直接断言"接口返回的解锁字段，等于数据库里存的那一行"——这才是真正该保证的东西。
